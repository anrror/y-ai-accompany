# AI情感Agent平台 — 需求分析文档

## 第一部分：产品起源与演进

### 1.1 v1 愿景：有温度的毛绒生命体（2025）

最初的设想是一个硬件产品——人工智能陪伴玩偶。

**核心洞察**：现代社会存在"结构性孤独"。人们购买的不是玩具，而是一个"永不背叛且完全懂你的倾听者"。市场上的智能音箱与毛绒玩具之间存在巨大断层——前者冰冷机械，后者沉默无声。

v1 定义了三大核心能力：
- **多模态仿生交互**：听觉（定向降噪方言识别）、触觉（柔性传感皮全场感知）、体温仿生（石墨烯均热板 35°C）、视觉微表情（高像素点阵眼）
- **三级记忆蒸馏网络**：短期工作区（Redis缓存）→ 情节总结区（T+1归档）→ 永久核心区（加密闪存）
- **主动关怀引擎**：情景触发 + 节假日记忆 + 生理节律感知

v1 的目标是"打造一个能与你共同呼吸、共同成长的实体灵魂搭子"。

### 1.2 市场验证与战略转折

在技术验证阶段，团队发现：

| 发现 | 影响 |
|------|------|
| 硬件 BOM 成本可控（约 ¥145），但开模/品控/供应链周期远超预期 | 硬件先行策略受阻 |
| 6 个 Agent 的 YAML 配置测试验证：同一 LLM 管线，不同配置产生完全不同的性格表现 | 平台化思路浮现 |
| 市场调研：98% 的潜在用户愿意尝试 AI 陪伴，但核心痛点是"没有个性化选择" | 多 Agent 平台是刚需 |
| 2026 年 4 月国家发布《人工智能拟人化互动服务管理暂行办法》 | 合规框架明确，平台化更利于统一管控 |

### 1.3 v2 重定义：从"一个硬件"到"一个平台"

**战略决策**：将产品从单一硬件转变为云原生 AI 情感 Agent 平台。

| 维度 | v1（硬件玩偶） | v2（Agent 平台） |
|------|---------------|-----------------|
| 形态 | 单一硬件设备 | 云平台托管多个 AI Agent |
| 交互 | 语音/触觉/温度 | 文本对话（可扩展至语音/视觉） |
| 记忆 | 本地闪存 | 云端 pgvector + Redis |
| 人格 | 固件预设 | Prompt 模板 + 性格引擎 |
| Agent 数量 | 固定 1 个 | 任意数量，独立身份 |
| 扩展性 | 固件升级 | API 注册，不改代码 |

**一句话定义**：AI情感Agent平台 = Agent 托管运行时 + 基础设施服务（记忆/性格/对话/安全）

#### I. 专用情感模型（HumanOmni-Emotion）

基于 HumanOmni 微调的专用情感模型，替代通用 LLM 兜底，将情感识别准确率推向量变级提升。

| ID | 功能 | 描述 | 优先级 |
|----|------|------|--------|
| F-I1 | 文本情感专用模型 | 基于 HumanOmni-7B LoRA 微调的 9 类情感分类模型，独立部署 | P0 |
| F-I2 | VAD 三维预测 | 专用模型直接输出 Valence/Arousal/Dominance 连续值，MAE < 0.05 | P0 |
| F-I3 | 配置化启用/降级 | 通过平台配置控制是否启用专用模型；未启用时自动降级为通用 LLM 兜底，零代码变更 | P0 |
| F-I4 | 数据标注与回流 | 构建平台标注数据集（DFEW/MAFW 映射 + GPT 合成 + 人工标注） | P0 |
| F-I5 | 三阶段微调+强化学习管线 | Phase A SFT 分类 → Phase B SFT VAD → Phase C SFT 对抗 → Phase D PPO RLHF | P0 |
| F-I6 | 模型量化部署 | 4-bit NF4 量化 + vLLM 推理，单次推理 < 50ms | P0 |
| F-I7 | 原生 Level 3 替换 | 替换 EmotionCenter + SentimentCenter 的 LLM 兜底层 | P0 |
| F-I8 | 多模态扩展接口 | 预留视频/音频/图片输入接口，支持 HumanOmni 原生多模态能力 | P1 |
| F-I9 | 模型版本管理与 A/B | 多版本模型在线 A/B 测试，灰度发布 | P1 |

**配置开关示例**：

```yaml
# 平台配置
emotion_detector:
  dedicated_model:
    enabled: true                          # true=使用专用模型, false=降级LLM兜底
    provider: "humanomni"                  # 专用模型Provider名称
    endpoint: "http://inference-cluster:8000/v1"
    model_name: "HumanOmni-Emotion-v1"
    fallback_on_error: true                # 推理失败时自动降级到LLM
```

| 指标 | 通用 LLM 兜底 | HumanOmni-Emotion |

| 隐性情感识别 | 不可靠 | ≥ 85% |

#### F. 情绪感知（双中心情感计算架构）

平台采用 **双中心并行架构**，将"情感"(Sentiment)和"情绪"(Emotion)作为两个独立的业务算法中心解耦设计：

| ID | 功能 | 描述 | 优先级 |
|----|------|------|--------|
| F-F1 | 情感极性检测 | SentimentCenter：识别消息的情感极性（正面/负面/中性）+ 强度分 [-1,1]，输出给性格演化模块 | P0 |
| F-F2 | 情绪类别检测 | EmotionCenter：识别 9 类离散情绪（喜悦/悲伤/愤怒/恐惧/惊讶/爱意/焦虑/感激/平静），输出 VAD 三维（Valence/Arousal/Dominance） | P0 |
| F-F3 | 三级检测流水线 | 每中心独立运行 Emoji(50+)→Keyword(60+)→LLM 三级渐进式检测，高置信度层命中即返回，避免不必要的 LLM 调用 | P0 |
| F-F4 | 滑动窗口情绪追踪 | 维护最近 N 轮情绪上下文，计算趋势（升/降/稳定）和主导情绪，注入 System Prompt | P0 |
| F-F5 | 情绪注入上下文 | 将 VAD 三维状态和情绪趋势注入 System Prompt，让 Agent 感知用户情绪状态 | P0 |
| F-F6 | 对话模式切换 | 根据用户状态切换 5 种对话模式（陪伴/反思/决策/恢复/成长） | P1 |

## 第六部分：数据模型概览

### 6.1 核心实体关系

```
用户（User）
  │
  ├── 1:N ── 会话（Session）
  │            │
  │            └── 1:N ── 消息（Message）
  │
  ├── N:M ── Agent ── 1:1 ── AgentConfig（身份、性格、能力、记忆配置）
  │            │
  │            └── N:M ── 用户 ──→ Memory（按 user_id + agent_id 隔离）
  │                        ├── WorkingMemory（最近 N 轮）
  │                        ├── ShortTermMemory（蒸馏摘要）
  │                        └── LongTermMemory（固化事实）
  │
  └── 1:1 ── UserProfile（年龄、紧急联系人、偏好设置）
```

### 6.2 数据隔离模型

```
隔离维度 = (user_id, agent_id)

user_1 × xiaonuan  → memory_namespace_1
user_1 × xingchen  → memory_namespace_2  （与 namespace_1 完全隔离）
user_2 × xiaonuan  → memory_namespace_3  （与 namespace_1 完全隔离）
```

Agent 之间不共享任何用户数据。用户之间不共享任何数据。
