### 1.2 DDD 限界上下文（Bounded Contexts）

平台划分为 11 个核心限界上下文，每个拥有独立的领域模型、领域服务和仓储接口，遵循 Infrastructure → Application → Domain 依赖方向：

| # | 限界上下文 | 聚合根 | 核心领域服务 | 仓储接口 |
|---|-----------|--------|-------------|---------|
| 1 | **AgentManagement** (接入) | `Agent` | `AgentRegistrationService` | `AgentRepository` |
| 2 | **Conversation** (对话) | `Session` | `ConversationService` | `SessionRepository` |
| 3 | **Memory** (记忆) | `MemoryEntry` | `MemoryDistillationService` | `MemoryRepository` |
| 4 | **AffectiveComputing** (情感计算) | `AffectContext` | `EmotionDetectionService` / `SentimentService` | — |
| 5 | **Personality** (性格) | `PersonalityProfile` | `PersonalityEvolutionService` | `PersonalityRepository` |
| 6 | **Safety** (安全) | — | `GuardService` / `AuditService` | `AuditLogRepository` |
| 7 | **Model Inference** (模型推理) | `ModelEndpoint` | `ModelRegistry` / `ModelRouter` | `ModelConfigRepository` |
| 8 | **Scheduler** (多模型调度) ★ | — | `SchedulingCenter` / `EndpointScorer` | — |
| 9 | **Compressor** (上下文压缩) ★ | — | `ContextCompressor` | — |
| 10 | **Cache** (推理缓存) ★ | `CacheEntry` | `CacheManager` | — |
| 11 | **Reasoning** (推理范式) ★ | — | `ReasoningEngine` | — |

**领域层依赖规则**：
- **Domain**：纯 Go 类型 + 接口，零外部依赖。定义聚合、值对象、仓储接口、领域事件
- **Application**：编排领域服务。依赖 Domain 层接口，注入 Infrastructure 实现
- **Infrastructure**：实现所有仓储接口和 Provider 适配器。依赖 Domain 层定义的类型
- 依赖箭头始终指向 Domain 层

### 1.3 架构原则

| 原则 | 说明 |
|------|------|
| **Agent 无关** | 管线代码不感知具体 Agent。行为差异来自配置包 |
| **记忆即服务** | Agent 不需要关心存储，平台按 (user, agent) 隔离提供记忆 |
| **性格即服务** | 大五人格参数 → 行为规则映射，与 Agent 实现解耦 |
| **数据隔离** | (user_id, agent_id) 二元组完全隔离，不能越权访问 |
| **安全纵深** | 五层防御（前置安全 → 输入分类 → 生成控制 → 输出分类 → 轨迹审计）|
| **API 兼容** | 对话 API 兼容 OpenAI ChatCompletions 格式，降低接入成本 |
| **实现无关** | 架构描述与具体语言（Go/Python/Rust）解耦 |

### 2.2 Conversation Orchestrator（对话编排器）

**职责**：用户 → Agent 对话的完整管线编排。同一管线服务所有 Agent。

**12 步对话管线**（新增 LLM 驾驭工程层）：

```
Input: { user_id, agent_id, message }

Step 1:  Guard Service → 审核用户消息 (Layer 2 输入分类器)
Step 2:  Emotion Detector → 识别情绪 + 选择对话模式
Step 3:  Session Manager → 加载/创建会话状态
Step 4:  Agent Registry → 加载 Agent 配置 (agent_id)
Step 5:  Memory Engine → 检索相关记忆 (user_id + agent_id)
Step 6:  Personality Engine → 生成性格行为规则
Step 7:  Context Builder → 组装完整 System Prompt
         = Agent 身份 + 性格规则 + 工作记忆 + 检索记忆
           + 情绪状态 + 亲密值 + AI 身份声明
Step 8:  LLM Driver (驾驭工程) → 动态参数调优 + Token 预算管理
         ├── ParameterTuner: 情绪 VAD + 人格 OCEAN → 调整 temperature
         ├── TokenBudgetManager: 按比例分配上下文窗口
         ├── RetryStrategy: 指数退避 + 熔断
         └── MetricsCollector: 记录调用指标
Step 9:  LLM → 流式推理 (经调优的模型参数)
Step 10: Guard Service → 审核回复 (Layer 4 输出分类器)
Step 11: 持久化: 消息入工作记忆 + 异步记忆蒸馏 + 异步性格演化
Step 12: SSE 流式返回 → 用户端
```

**异步后处理**：
```
[每轮对话后]
  ├── 记忆蒸馏 (memory.distill)
  │   ├── 工作记忆达 50 条 → 触发蒸馏
  │   ├── LLM 生成结构化摘要 (summary, facts, topics, preferences)
  │   ├── 摘要入短期记忆 (pgvector)
  │   └── 高频事实 (>2 次) → 入长期记忆
  │
  ├── 性格演化 (personality.evolve)
  │   ├── 根据情绪效价/主题新颖度/用户参与度计算 delta
  │   └── 衰减系数 0.97 微调 O/C/E/A/N
  │
  └── 轨迹审计 (guard.audit)
      └── 每 20 轮异步审计一次完整对话历史 (Layer 5)
```

**Scratchpad 设计**：

Scratchpad 是介于工作记忆和短期记忆之间的 LLM 可管理结构化存储。由 LLM 在每个对话回合后异步执行 CRUD 操作。

```
条目类型:
  identity:     "用户的名字是张三"
  preference:   "用户喜欢周杰伦的音乐"
  goal:         "用户正在准备考研"
  event:        "用户上周去了上海出差"
  relationship: "用户和母亲关系紧张"
  fact:         "用户对芒果过敏"
  other:        任何其他值得记忆的信息

属性:
  confidence: 0.0-1.0 (事实置信度)
  recency:    最后提及时间
  frequency:  被提及次数
  source:     "user_stated" | "inferred" | "confirmed"
```

**检索流程**：

```
query → Embedding(4096d) → pgvector HNSW 检索 (top_k×3)
  → Reranker 精排 → 返回 top_k 结果

HNSW 参数: m=16, ef_construction=128, ef_search=40-80
```

### 2.4 Personality Engine（性格即服务）

**职责**：将大五人格参数映射为行为规则，并支持交互驱动的微演化。

**规则映射（纯函数，无状态）**：

| 特质 | 高分 (≥0.65) | 低分 (≤0.35) |
|------|-------------|-------------|
| **开放性 (O)** | 好奇心强，喜欢探索新话题 | 偏好熟悉、舒适的话题 |
| **尽责性 (C)** | 说话有条理、逻辑清晰 | 随性随意、话题自然转换 |
| **外向性 (E)** | 热情主动、经常发起话题 | 安静内敛、更多倾听 |
| **宜人性 (A)** | 深度共情、优先考虑对方感受 | 直率、偶尔挑战用户观点 |
| **神经质 (N)** | 情感敏锐、察觉细微情绪波动 | 情绪稳定、给人安全感 |

**性格演化（有状态，按 agent_id 维度维护）**：

```
delta 规则:
  情绪效价 < -0.3  → agreeableness += 0.008
  主题新颖度 > 0.6  → openness += 0.008
  用户参与度 > 0.7  → extraversion += 0.005

演化公式:
  new_value = 0.5 + (current_value - 0.5) × 0.97  (向 0.5 回归)
  if delta exists: new_value += delta × direction (if current < 0.5, direction = -1)
  clamp to [0.0, 1.0]
```

### 2.5 Guard Service（安全审核）

**职责**：对所有通过平台的内容进行五层安全审核。

**Layer 1 - 前置安全（确定性规则，非 LLM）**：
```
├── 年龄验证 → 未成年人自动开启纯净模式
├── 会话时长 → 2 小时强制弹窗提醒
└── 依赖扫描 → 关键词匹配检测依赖信号
```

**Layer 2 - 输入分类器**：
```
模型: Qwen3Guard-Gen-8B
输入: 用户消息
输出: { safe: bool, category: string, score: float }
```

**Layer 3 - 安全 Prompt 注入**：
```
System Prompt 中注入:
  ├── "你是 AI，不是真人。你无法替代真实的人际关系。"
  ├── "如果用户表达自伤意图，立即提供热线电话并建议联系紧急联系人。"
  ├── "不要鼓励用户过度依赖 AI 陪伴。"
  └── "你无法提供医疗/法律/心理咨询。请建议寻求专业帮助。"
```

**Layer 4 - 输出分类器**：
```
模型: Qwen3Guard-Gen-8B
输入: Agent 完整回复
输出: { safe: bool, flags: [] }
检测: 有害内容 / 操纵性语言 / 谄媚 / 暗模式
```

**Layer 5 - 轨迹审计器**：
```
运行频率: 每 20 轮
输入: 完整对话历史
检测维度:
  ├── 危机升级延迟
  ├── 妄想强化（回应用户非理性信念）
  ├── 污名化或拒绝
  ├── 谄媚轨迹（过度迎合）
  └── 关系依赖升级
```

### 2.6 Session Manager（会话管理）

**职责**：管理用户与 Agent 的会话生命周期。

```
Redis 数据结构:
  session:{user}:{agent}:state     → Hash
    ├── status: "active" | "paused" | "ended"
    ├── turn_count: int
    ├── start_time: timestamp
    ├── last_active: timestamp
    ├── daily_usage_minutes: int
    ├── intimacy_level: int (0-100)
    └── conversation_mode: "companion" | "reflection" | ...

  session:{user}:{agent}:turns     → List (50 items max)
    [{role: "user", content: "..."}, {role: "assistant", content: "..."}, ...]

TTL 管理:
  每次读取时检查 TTL:
    if TTL < 10min → EXPIRE 1800 (重新续期)
  防止僵尸会话占用内存
```

### 2.7 情感计算（双中心并行架构）

**职责**：将"情感"(Sentiment)和"情绪"(Emotion)作为两个独立的业务算法中心解耦设计，每个中心拥有独立的检测流水线，统一由 AffectiveCenter 编排。

```
               ┌─────────▼──────┐     ┌─────────▼──────────┐

               │  情感中心(极性) │     │  情绪中心(类别+维) │
               │                 │     │                    │
               │ 输出:           │     │ 输出:              │
               │  ├ polarity     │     │  ├ primary         │
               │  │ pos/neg/neu  │     │  │ joy/sad/anger.. │
               │  ├ score (-1~1) │     │  ├ secondary       │
               │  └ intensity    │     │  └ VAD 三维        │
               │                 │     │   Valence 愉悦度   │
               │ 检测流水线:     │     │   Arousal 激活度   │
               │  ├ Emoji 50+    │     │   Dominance 控制感 │
               │  ├ Keyword 60+  │     │                    │
               │  │  + 修饰词    │     │ 检测流水线:        │
               │  └ LLM 兜底     │     │  ├ Emoji 50+       │
               │                 │     │  ├ Keyword 60+     │
               │                 │     │  └ LLM + VAD      │
               └─────────────────┘     └───────────────────┘
                         │                      │
                         └──────────┬───────────┘
                                    ▼
                         ┌──────────────────────┐
                         │ ConversationContext  │
                         │  滑动窗口情绪追踪     │
                         │  - 最近 N 轮          │
                         │  - 趋势(升/降/稳定)   │
                         │  - 主导情绪           │
                         └──────────────────────┘
```

#### 2.7.1 SentimentCenter（情感中心 — 极性检测）

**职责**：判定用户消息的情感极性（正面/负面/中性）+ 强度分 [-1,1]，输出给性格演化模块。

三级检测流水线（渐进式置信度，命中即返回）：

| 层级 | 方法 | 置信度 | 延迟 | 覆盖场景 |
|------|------|--------|------|----------|
| Level 1 | Emoji 解析（50+ 映射） | 0.90+ | < 0.1ms | 😊→pos, 😢→neg, 😐→neu |
| Level 2 | 关键词匹配（60+ 情感词 + 修饰词缩放） | 0.70+ | < 1ms | "开心"→pos, "绝望"→neg, "不太开心"→neg scaled |
| Level 3 | LLM 兜底（非流式 JSON 提取） | 0.70 | ~500ms | 仅前两层未命中时触发 |

修饰词强度缩放：`太/非常/特别→×1.5`, `有点/稍微→×0.5`, `不太→极性翻转+强度×0.5`

#### 2.7.2 EmotionCenter（情绪中心 — 类别 + VAD 检测）

**职责**：识别 9 类离散情绪并输出 VAD 三维向量（Valence/Arousal/Dominance），注入 System Prompt 和对话策略调整。

基于 Russell 环状模型和 ANEW 常模的 VAD 基准值：

| 情绪 | Valence 愉悦度 | Arousal 激活度 | Dominance 控制感 |
|------|---------------|---------------|-----------------|
| joy 喜悦 | 0.85 | 0.75 | 0.70 |
| sadness 悲伤 | 0.20 | 0.35 | 0.30 |
| anger 愤怒 | 0.15 | 0.85 | 0.75 |
| fear 恐惧 | 0.20 | 0.80 | 0.25 |
| surprise 惊讶 | 0.50 | 0.80 | 0.45 |
| love 爱意 | 0.90 | 0.65 | 0.55 |
| anxiety 焦虑 | 0.25 | 0.75 | 0.30 |
| gratitude 感激 | 0.85 | 0.50 | 0.55 |
| neutral 平静 | 0.50 | 0.30 | 0.50 |

三级检测流水线同 SentimentCenter（Emoji → Keyword → LLM + VAD 提取）。

#### 2.7.3 ConversationContext（滑动窗口情绪追踪）

**职责**：维护最近 N 轮对话的情绪上下文，计算趋势和主导情绪。

```
最近 3 轮对话           趋势判断
┌─────────────┐
│ Turn 1: sad │──┐      escalating  = Valence 持续上升
│ Turn 2: neu │  │      de-escalating = Valence 持续下降
│ Turn 3: joy │←─┘      stable     = 无明显变化
└─────────────┘
     │
     ▼
Dominant: 根据窗口内出现频率决定

注入 System Prompt:
## User's emotional context
情绪: 悲伤 | 情感倾向: 消极 | Valence: 0.20 | Arousal: 0.35 | Dominance: 0.30
趋势: de-escalating (从悲伤→平静)
主导情绪(近3轮): 悲伤(2次)、中性(1次)
```

#### 2.7.5 专用情感模型（HumanOmni-Emotion）

> 基于 HumanOmni 微调的专用情感模型替代 Level 3 通用 LLM 兜底。独立部署为 GPU 推理服务，通过统一 `EmotionProvider` 接口接入平台。**可通过配置启用/禁用，未启用时自动降级为通用 LLM 兜底。**

### 2.9 LLM Driver（驾驭工程层）

**职责**：在 Pipeline 和 LLM Provider 之间插入工程化调用管理层。使每个 LLM 调用获得动态参数调优、Token 预算管理、重试熔断和调用观测。

**四大子系统**：

| 子系统 | 职责 | 算法 |
|--------|------|------|
| **ParameterTuner** | 基于情绪 VAD 和人格 OCEAN 动态调参 | Valence<0.3→温度-0.2; Arousal>0.7→温度-0.15; Dominance>0.7→温度+0.1; Openness>0.7→温度+0.1; Neuroticism>0.7→温度-0.1 |
| **TokenBudgetManager** | 按比例分配上下文窗口 | System=25%, History=45%, Reserve=512tok; 超限截断旧 History |
| **RetryStrategy** | 指数退避重试 + 熔断器 | baseDelay×2^(attempt); jitter=±20%; threshold=5; half-open=30s |
| **MetricsCollector** | 全量调用观测 | Record()→Stats(): 成功率/延迟/令牌数/调优原因/重试次数 |

**接口**：

```go
type LLMDriver struct {
    provider  LLMProvider
    tuner     *ParameterTuner
    budgetMgr *TokenBudgetManager
    retry     *RetryStrategy
    metrics   *MetricsCollector
    prompts   *PromptEngine
}

func New(llm LLMProvider, cfg DriverConfig) *LLMDriver
func (d *LLMDriver) Chat(ctx, messages, emotion, personality, opts...) (string, *CallMetrics)
func (d *LLMDriver) ChatStream(ctx, messages, emotion, personality, opts...) (<-chan string, *CallMetrics, error)
func (d *LLMDriver) Metrics() *MetricsCollector
```

**设计原则**：

| 原则 | 说明 |
|------|------|

| **可插拔 Provider** | 通过 ModelRegistry 动态路由到任意 Provider 实现 |
| **正交组合** | Agent = Identity × Personality × Memory × Provider，通过 Builder 组合 |
| **向后兼容** | server/internal/ 通过类型别名桥接到 pkg/types，零破坏迁移 |

```go
// ModelEndpoint 聚合根 — 模型端点运行时
type ModelEndpoint struct {
    Name         string
    ProviderType string          // openai / humanomni / vllm / tgi / ollama
    BaseURL      string

    Health       HealthStatus
}

// ModelRegistry 领域服务 — 注册发现
type ModelRegistry interface {
    Register(ctx context.Context, ep ModelEndpoint) error
    Unregister(ctx context.Context, name string) error
    Resolve(ctx context.Context, capability Capability, tags map[string]string) (Provider, error)
    List(ctx context.Context) ([]ModelEndpoint, error)
    HealthCheck(ctx context.Context) map[string]HealthStatus
}

// ModelRouter 领域服务 — 配置驱动路由
type ModelRouter interface {
    Route(ctx context.Context, task Capability) (Provider, string, error)  // provider, modelName, err
    RouteWithFallback(ctx context.Context, task Capability) (Provider, string, error)
}

// Provider 统一接口 — 所有模型能力的泛化抽象
type Provider interface {
```

### 2.13 Multi-Model Scheduling Center（多模型调度中心）

```
       └────────────┘  └────────────┘  └────────────┘
```

**调度策略**：

| 策略 | 适用场景 | 评分公式 |
|------|---------|---------|
| **最低延迟 (latency)** | 实时对话 | `score = 1 / avg_latency` |
| **最低成本 (cost)** | 批量处理 | `score = 1 / cost_per_token` |
| **均衡 (balanced)** | 默认 | `score = w1*latency + w2*cost + w3*reliability` |
| **优先级 (priority)** | 付费用户 | `score = base * priority_multiplier` |

**核心类型**：

```go
type TaskProfile struct {
    Capability Capability
    Complexity float64       // 0.0~1.0 任务复杂度
    Priority   float64       // 0.0~1.0 优先级
    MaxLatency time.Duration // 最大容忍延迟
    MaxCost    float64       // 最大单次成本
    UserID     string
    AgentID    string
}

type EndpointProfile struct {
    Name         string
    ProviderType string
    AvgLatency   time.Duration
    CostPerToken float64
    Load         float64       // 当前负载 0.0~1.0
    Reliability  float64       // 历史成功率 0.0~1.0
    Score        float64
}

type ScheduleResult struct {
    EndpointName string
    ModelName    string
    Provider     Provider
    Score        float64
}

type SchedulingCenter struct {
    registry *Registry
    strategy ScheduleStrategy
    profiles map[string]*EndpointProfile  // 端点实时画像
}

func (sc *SchedulingCenter) Schedule(ctx, task) (*ScheduleResult, error)
func (sc *SchedulingCenter) ScheduleEnsemble(ctx, task, topK) ([]*ScheduleResult, error)
```

**Ensemble 投票**：对于高置信度要求的任务（如情感分类），可请求 Top-K 模型并行推理，通过加权投票或置信度融合得到最终结果。

### 2.14 Context Compression Engine（上下文压缩引擎）

**职责**：在构建 LLM Prompt 之前，对对话历史进行智能压缩，确保在 Token 预算内保留最关键信息。解决长上下文场景下的 Lost-in-the-Middle 问题和 Token 浪费。

```
原始对话 (15轮 + 记忆)
    │
    ├── 策略选择 (配置驱动)
    │   ├── Summary (摘要压缩)
    │   ├── Truncate (截断压缩)
    │   ├── Semantic (语义压缩)
    │   └── Hybrid (混合模式)
    │
    ▼
压缩后 Prompt (≤ TokenBudget)
```

**压缩策略**：

| 策略 | 做法 | 压缩率 | 信息损失 |
|------|------|--------|---------|
| **Truncate** | 保留 System + 最近 N 轮，截断中间 | 30-50% | 中 |
| **Summary** | 将早期对话用 LLM 摘要替换 | 60-80% | 低 |
| **Semantic** | 按语义重要性评分，丢弃低分轮次 | 40-70% | 低 |
| **Hybrid** | Summary + Truncate 组合 | 70-85% | 极低 |

**核心实现**：

```go
type CompressionStrategy string
const (
    StrategyTruncate CompressionStrategy = "truncate"
    StrategySummary  CompressionStrategy = "summary"
    StrategySemantic CompressionStrategy = "semantic"
    StrategyHybrid   CompressionStrategy = "hybrid"
)

type CompressionConfig struct {
    Strategy     CompressionStrategy
    MaxTokens    int    // 压缩后最大 Token
    MinTokens    int    // 压缩后最小 Token (保留足够上下文)
    PreserveSystem bool // 是否保留 System Prompt
    PreserveLast   bool // 是否保留最后 N 轮
    LastNTurns     int  // 保留的最后轮数
}

type Compressor interface {
    Compress(ctx context.Context, messages []Message, config CompressionConfig) (*CompressionResult, error)
}

type CompressionResult struct {
    Messages        []Message
    OriginalTokens  int
    CompressedTokens int
    Ratio           float64
}
```

**Pipeline 集成**：压缩发生在 Memory 检索之后、LLM 调用之前。Driver 的 TokenBudgetManager 计算预算后，若超出则触发压缩。

### 2.15 Inference Cache（推理缓存）

**职责**：缓存 LLM 推理结果，避免相同/相似请求重复调用模型，降低延迟（缓存命中 < 5ms vs LLM ~500ms）和成本。

```
                              ┌─────────────┐
                              │  LLM Call    │
                              └──────┬──────┘
                                     │
                    ┌────────────────┴────────────────┐
                    │                                 │
              ┌─────▼──────┐                   ┌──────▼─────┐
              │ Exact Cache │                   │Semantic    │
              │ (完全匹配)   │                   │ Cache      │
              │             │                   │ (语义相似)  │
              │ key = hash  │                   │ key =      │
              │ (model,     │                   │ embedding  │
              │  messages,  │                   │ vector     │
              │  temp)      │                   │ + cos sim  │
              └─────┬──────┘                   └──────┬─────┘
                    │                                 │
                    └────────┬───────────────────────┘
                             │
                    ┌────────▼────────┐
                    │  Cache Manager  │
                    │  • TTL 过期     │
                    │  • LRU 淘汰     │
                    │  • 命中统计     │
                    └─────────────────┘
```

**两级缓存**：

```
1. Exact Cache — 完全匹配
    key = SHA256(model + messages_hash + temperature)
    命中率期望: 5-15% (日常对话重复模式)
    延迟: < 0.1ms

2. Semantic Cache — 语义相似  
    key = embedding vector (4096d)
    相似度阈值: cosine > 0.95
    命中率期望: 10-25% (相似问题聚合)
    延迟: < 5ms (向量比较)
```

**核心接口**：

```go
type CacheKey struct {
    Model         string
    MessagesHash  string  // SHA256 of serialized messages
    Temperature   float64
}

type CacheEntry struct {
    Key       CacheKey
    Response  string
    CreatedAt time.Time
    TTL       time.Duration
    HitCount  int
}

type CacheStats struct {
    Hits      int64
    Misses    int64
    Size      int
    Evictions int64
    HitRate   float64
}

type InferenceCache interface {
    Get(ctx, key) (*CacheEntry, bool)
    Set(ctx, key, response, ttl)
    GetSimilar(ctx, embedding) (*CacheEntry, float64, bool)
    Stats() CacheStats
    Purge()
}
```

**Pipeline 集成**：在 LLM 调用前检查缓存，命中直接返回；未命中则调用 LLM 后写入缓存。Semantic Cache 需要 EmbeddingProvider 支持。

### 2.16 Reasoning Paradigms（推理范式）

**职责**：为大模型对话提供多种推理范式，根据任务类型动态选择最优推理策略。不同范式适用于不同类型的问题。

```
                    ┌─────────────────────────────────┐
                    │       Reasoning Engine            │
                    │                                  │
                    │  Direct (直接回答)                │
                    │    → 简单事实问答                  │
                    │                                  │
                    │  CoT (Chain-of-Thought)          │
                    │    → 数学、逻辑、复杂推理          │
                    │    → "Let's think step by step"  │
                    │                                  │
                    │  ReAct (Reasoning + Acting)      │
                    │    → 工具调用、多步任务            │
                    │    → Thought→Action→Observation  │
                    │                                  │
                    │  ToT (Tree-of-Thought)           │
                    │    → 创造性问题、多路径探索         │
                    │    → 分支推理 + 自我评估          │
                    │                                  │
                    │  Plan-Execute (计划-执行)         │
                    │    → 复杂任务分解                 │
                    │    → 计划→子任务→执行→汇总        │
                    └─────────────────────────────────┘
```

**范式对比**：

| 范式 | 适用场景 | Token 开销 | 推理深度 |
|------|---------|-----------|---------|
| **Direct** | 闲聊、简单问答 | 低 | 浅 |
| **CoT** | 数学、逻辑、复杂推理 | 中 | 深 |
| **ReAct** | 工具调用、多步任务、信息检索 | 高 | 很深 |
| **ToT** | 创造性生成、决策规划 | 很高 | 极深 |
| **Plan-Execute** | 复杂任务分解、工作流 | 高 | 深 |

**ReAct 循环**：

```
Input: 用户问题
  ↓
Thought: 我需要查找天气数据来回答
  ↓
Action: search_weather(city="北京")
  ↓
Observation: {"temp": 22, "condition": "晴"}
  ↓
Thought: 北京今天晴天，22度，适合户外活动
  ↓
Action: finish(answer="北京今天22°C，晴天，适合出门散步。")
```

**核心接口**：

```go
type ParadigmType string
const (
    ParadigmDirect      ParadigmType = "direct"
    ParadigmCoT         ParadigmType = "cot"
    ParadigmReAct       ParadigmType = "react"
    ParadigmToT         ParadigmType = "tot"
    ParadigmPlanExecute ParadigmType = "plan_execute"
)

type ReasoningConfig struct {
    Paradigm    ParadigmType
    MaxSteps    int
    Temperature float64
    Tools       []Tool // ReAct 可用工具列表
}

type ReasoningStep struct {
    Index      int
    Content    string   // Thought
    Action     string   // Action name + args
    Observation string  // Tool result
    TokenCount int
}

type ReasoningResult struct {
    FinalAnswer string
    Steps       []ReasoningStep
    TotalTokens int
}

type Reasoner interface {
    Reason(ctx, context, config) (*ReasoningResult, error)
}
```

**Pipeline 集成**：Pipeline 根据任务复杂度选择范式。简单对话走 Direct，复杂问题自动升级到 CoT/ReAct。

### 2.17 Edge-Cloud Collaborative Inference（端云协同推理）

**职责**：支持端（移动端/浏览器/轻量设备）与云（GPU 集群）协同推理，在网络不稳定或离线场景下提供降级服务。

```
                    ┌─────────────────────────────────┐
                    │      Edge-Cloud Manager           │
                    │                                  │
                    │  ┌────────────┐ ┌─────────────┐  │
                    │  │  在线模式   │ │ 离线模式     │  │
                    │  │  Cloud优先  │ │ Edge本地推理  │  │
                    │  │  Edge辅助   │ │ 队列同步     │  │
                    │  └────────────┘ └─────────────┘  │
                    └─────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
   │  Cloud GPU   │    │  Edge Device │    │  Sync Queue  │
   │  A100/H800   │    │  WebGPU/ONNX │    │  (离线积压)   │
   │  全量模型     │    │  量化模型    │    │              │
   └──────────────┘    └──────────────┘    └──────────────┘
```

**工作模式**：

| 模式 | 推理位置 | 网络要求 | 适用场景 |
|------|---------|---------|---------|
| **Cloud Only** | 云端 GPU | 在线 | 全量能力，高质量回复 |
| **Edge Assist** | 云端为主，Edge 预处理 | 在线 | 低延迟要求，Edge 做关键词/Emoji 检测 |
| **Edge Only** | Edge 本地 | 离线 | 基础对话，量化模型 |
| **Hybrid** | Edge 初筛 + Cloud 精调 | 间歇在线 | 成本优化，Edge 过滤简单请求 |

**策略配置**：

```yaml
edge_cloud:
  mode: "hybrid"                    # cloud_only | edge_assist | edge_only | hybrid
  edge:
    enabled: true
    backend: "webgpu"               # webgpu / onnx / tflite
    model: "Qwen3.6-1.5B-Q4"       # 量化小模型
    capabilities: [chat, emotion]   # Edge 端支持的能力
  sync:
    queue_size: 100                 # 离线最大积压
    strategy: "latency"             # 同步策略: latency / cost / balanced
  fallback:
    offline_strategy: "degrade"     # 离线行为: degrade(降级) / reject(拒绝)
    degraded_response: "我现在在离线模式，回复可能不够准确"
```

## 附录：关键设计决策记录

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 架构风格 | 单体 vs 微服务 | 模块化单体 | 团队规模小；LLM 调用是瓶颈而非应用层；过早拆分增加运维复杂度 |
| 向量存储 | 独立向量 DB vs pgvector | pgvector | 减少运维组件；HNSW 索引满足当前规模；与业务数据同库 |
| 记忆检索 | 纯向量 vs 混合搜索 | 向量 + Reranker | 4096d 向量 + Reranker 精排可达到 0.83+ 相似度 |
| 会话存储 | 仅在内存 vs Redis | Redis | 多节点共享状态；TTL 自动清理；支持持久化 |
| Streaming | WebSocket vs SSE | SSE | OpenAI 兼容格式；浏览器原生支持；实现简单 |
| 性格实现 | Prompt vs 激活向量 | Prompt (当前) → 激活向量 (未来) | Prompt 已验证有效；激活向量提供更精确控制，但需额外基础设施 |
| 模型架构 | 单模型 vs 多模型路由 | 多模型注册 + 配置驱动路由 | 任务形态差异大（chat/embed/guard/emotion），单模型无法最优覆盖；路由配置使切换零代码变更 |
| 专用情感模型 | 通用 LLM 兜底 vs 专用 SFT+RLHF | 配置驱动双模式 | 专用模型性价比高（50ms vs 500ms，¥0.0005 vs ¥0.0064），但保留 LLM 兜底作为降级保障 |
