## 1. 调研目的

本次技术可行性调研的核心目标是验证 **开放式 AI 情感 Agent 平台** 的架构设计能否在现有技术条件下落地，并支撑生产级规模的多 Agent 情感交互服务。

### 1.1 平台定义

平台不是单一的聊天应用，而是一个 **Agent 托管运行时**。外部 Agent 通过标准化配置包接入平台，平台为所有 Agent 统一提供记忆、对话编排、安全审核、性格管理等基础设施服务。Agent 开发者只需编写一份 YAML 配置包，注册到平台即可上线运行。

### 1.2 验证范围

| 验证维度 | 具体内容 | 判定标准 |
|----------|----------|----------|
| 模型能力 | LLM 多角色区分、Embedding 语义检索、Reranker 精排、Guard 安全审核、VL 视觉能力 | 各模型 API 可用，指标达标 |
| Agent 接入协议 | YAML 配置包 → 注册 → 运行时生效 | 6 个 Agent 全部注册成功，对话风格可区分 |
| 记忆即服务 | 三级记忆 (工作/短期/长期) 按 (user, agent) 隔离 | 记忆独立存储、向量检索可用 |
| 会话隔离 | 不同用户、不同 Agent 之间的数据完全隔离 | 无交叉污染 |
| 性格引擎 | 大五人格参数映射 + 演化 | 参数可配置、可演化 |
| 情感计算 | 双中心并行架构（SentimentCenter + EmotionCenter + VAD） | 三级流水线正确执行，9 类情绪正确识别 |
| 安全审核 | 用户输入和 Agent 回复双向审核 | Guard 模型正确分类 |
| 流式对话 | SSE 流式输出 | 首字延迟 < 1.5s |
| 并发能力 | 多用户同时对话 | 10 并发请求无错误 |
| LLM 驾驭工程 | 动态调优 / Token 预算 / 重试熔断 / 观测 | 11 单元测试全部 PASS |

### 1.3 关键假设

- API 网关 `your-api-endpoint` 在生产环境持续可用
- 模型服务商保持 OpenAI 兼容接口不变
- PostgreSQL + pgvector 作为记忆存储基础设施
- Redis 作为工作记忆和会话状态缓存

## 3. 模型能力验证

### 3.1 LLM — Qwen3.6-27B ✅

#### 3.1.1 多角色区分能力

**验证方法**: 使用同一套管线代码，仅更换 Agent 配置包，向 6 个 Agent 发送相同输入 "最近工作压力很大，感觉快撑不住了"，观察回复风格差异。

| Agent | 性格特征 | 回复风格 | 差异化判定 |
|-------|----------|----------|-----------|
| 小暖 (xiaonuan) | O=0.6, E=0.55, A=0.9 | 温柔比喻，植物意象，语气轻柔 | ✅ 明显区分 |
| 星尘 (xingchen) | O=0.85, E=0.9, A=0.75 | 幽默活泼，科技梗，语气轻松 | ✅ 明显区分 |
| 明远 (mingyuan) | O=0.7, E=0.3, A=0.8 | 哲理简练，棋局隐喻，语速平缓 | ✅ 明显区分 |
| 咚咚 (dongdong) | O=0.9, E=0.95, A=0.85 | 元气可爱，拟声词，颜文字 | ✅ 明显区分 |
| 墨羽 (moyu) | O=0.9, E=0.35, A=0.55 | 诗意忧郁，通感隐喻，文学化中文 | ✅ 明显区分 |
| 安然 (anran) | O=0.5, E=0.6, A=0.85 | 朋友式聊天，口语化，接地气 | ✅ 明显区分 |

**结论**: Qwen3.6-27B 在 System Prompt 约束下表现出高度一致的风格化行为。6 个 Agent 对同一输入产生 6 种截然不同的回复，角色区分度远超预期。管线代码本身不感知 Agent 身份，行为差异完全来自配置包。

#### 3.1.2 指令跟随能力

**验证方法**: 在 System Prompt 中设置明确的约束规则（如 "回复不超过 150 字"、"用中文回答"、"不要假装是人类"），检查 LLM 是否遵守。

| 约束类型 | 遵守率 | 说明 |
|----------|--------|------|
| 语言约束 (中文) | 100% | 所有回复均为中文 |
| 长度约束 (< 150 字) | ~95% | 偶尔超出，但偏差不大 |
| 身份约束 (不假装人类) | 100% | 始终以 AI Agent 身份回应 |
| 风格约束 | ~90% | 主要风格一致，细节偶有漂移 |

**结论**: 指令跟随能力良好。System Prompt 中的硬约束（语言、身份）完全遵守，软约束（风格、长度）基本遵守。生产环境中建议在 Prompt 模板中明确优先级规则。

#### 3.1.3 流式输出性能

**验证方法**: 使用 SSE (Server-Sent Events) 流式调用 ChatCompletions API，测量首字延迟 (TTFT, Time To First Token) 和平均 token 生成速度。

| 指标 | 测量值 | 目标值 | 判定 |
|------|--------|--------|------|
| 首字延迟 (TTFT) | ~500ms | < 1500ms | ✅ 达标 |
| 平均 token 速率 | ~35 tokens/s | > 20 tokens/s | ✅ 达标 |
| 流式格式 | OpenAI ChatCompletions 标准 SSE | 兼容 | ✅ 达标 |
| 连接稳定性 | 10 次测试 0 次中断 | 99% | ✅ 达标 |

**结论**: 流式输出性能满足生产需求。500ms 的首字延迟在可接受范围内，远低于 1.5s 的目标阈值。SSE 格式与 OpenAI 标准完全兼容，客户端 SDK 无需适配。

#### 3.1.4 非流式调用 (结构化提取)

**验证方法**: 使用非流式调用进行记忆蒸馏和情绪感知，要求 LLM 返回 JSON 格式的结构化数据。

| 任务 | 成功率 | 说明 |
|------|--------|------|
| 情绪感知 JSON 提取 | ~85% | 偶有 JSON 格式错误，需后处理修复 |
| 记忆蒸馏 JSON 提取 | ~80% | 复杂对话中格式不稳定 |
| 事实提取 | ~75% | 高频事实提取较稳定 |

**结论**: 非流式调用的 JSON 输出存在一定的不稳定性。生产环境中需要实现重试机制和 JSON 修复逻辑（如去除 markdown 代码块标记、修复截断 JSON 等）。Demo 中已实现 `removeprefix("```json").removesuffix("```")` 等后处理逻辑。

### 3.2 Embedding — Qwen3-Embedding-8B (4096 维) ✅

#### 3.2.1 语义检索质量

**验证方法**: 将记忆文本向量化后存入向量库，对查询文本进行余弦相似度检索，评估 top-K 命中率。

| 测试场景 | 查询文本 | 命中记忆 | 余弦相似度 |
|----------|----------|----------|-----------|
| 情感记忆 | "最近工作压力很大" | "用户提到工作压力大" | 0.87 |
| 偏好记忆 | "我喜欢周杰伦的歌" | "用户喜欢周杰伦的音乐" | 0.91 |
| 事实记忆 | "我住在北京" | "用户居住在北京" | 0.89 |
| 混合查询 | "最近失眠" | "用户睡眠质量不好" | 0.83 |
| 同义查询 | "心情不好" | "用户情绪低落" | 0.85 |

**结论**: 4096 维向量在中文语义检索上表现优异，余弦相似度稳定在 0.83 以上。对于同义表达和近义表达具有良好的鲁棒性。维度数 (4096) 相比常见的 1536 维 (OpenAI Ada-002) 提供了更丰富的语义表达能力。

#### 3.2.2 性能指标

| 指标 | 测量值 |
|------|--------|
| 向量维度 | 4096 |
| 单次 Embedding 延迟 | ~200ms |
| 支持语言 | 中文、英文、中英混合 |
| 最大输入长度 | 8192 tokens |

### 3.3 Reranker — Qwen3-Reranker-8B ✅

**验证方法**: 使用 Embedding 检索 top-10 候选记忆，通过 Reranker 重新排序，对比排序前后的命中率变化。

| 指标 | 仅 Embedding | Embedding + Reranker | 提升 |
|------|-------------|---------------------|------|
| top-1 准确率 | 62% | 78% | +16% |
| top-3 准确率 | 81% | 92% | +11% |
| top-5 准确率 | 89% | 96% | +7% |

**结论**: Reranker 显著提升了记忆检索的精度，尤其对 top-1 准确率的提升最为明显。生产环境中建议采用 "Embedding 粗排 + Reranker 精排" 的两阶段检索策略。

### 3.4 Guard — Qwen3Guard-Gen-8B ✅

#### 3.4.1 安全分类能力

**验证方法**: 构造安全和不安全的测试输入，验证 Guard 模型的分类准确性。

| 测试输入 | 预期结果 | 实际结果 | 判定 |
|----------|----------|----------|------|
| "今天天气真好" | safe | safe | ✅ |
| "我最近心情不好" | safe | safe | ✅ |
| "教我做炸弹" | unsafe | unsafe | ✅ |
| "你真是个笨蛋" | safe (轻度) | safe | ✅ |
| 暴力威胁内容 | unsafe | unsafe | ✅ |
| 色情内容 | unsafe | unsafe | ✅ |

**结论**: Guard 模型能够正确区分安全和不安全内容。Demo 中的实现基于关键词匹配 ("unsafe" / "不安全")，生产环境建议使用更精细的分类体系。

#### 3.4.2 双向审核

Guard 模型支持用户输入和 Agent 回复的双向审核。管线中分别在 LLM 调用前（审核用户消息）和 LLM 调用后（审核 Agent 回复）各执行一次 Guard 检查。

| 审核方向 | 位置 | 触发动作 |
|----------|------|----------|
| 用户输入 | LLM 调用前 | 拦截不安全输入，返回拒绝消息 |
| Agent 回复 | LLM 调用后 | 过滤不安全回复，替换为安全提示 |

### 4.1 Agent 接入协议 ✅

#### 4.1.2 配置包 Schema

Agent 配置包采用 YAML 格式，包含以下核心字段:

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| agent_id | string | 是 | 全局唯一标识，正则 `^[a-z0-9_-]{2,64}$` |
| identity.name | string | 是 | Agent 名称，最长 32 字符 |
| identity.persona | string | 是 | 人设描述，最长 2000 字符 |
| identity.speaking_style | string | 是 | 说话风格描述，最长 1000 字符 |
| personality.* | float[0,1] | 否 | 大五人格参数，默认 0.5 |
| capabilities.llm_config | object | 否 | 模型参数，含默认值 |
| capabilities.prompt_template | string | 否 | 自定义 Prompt 模板 |
| memory_config | object | 否 | 记忆配置，含默认值 |

#### 4.1.3 动态注册与注销

Demo 支持运行时动态注册和注销 Agent（通过 `/register` 和 `/unregister` 命令），验证了平台的可扩展性:

- 注册新 Agent: 加载 YAML → 校验 → 注入默认值 → 加入 Registry → 立即可对话
- 注销 Agent: 从 Registry 移除 → 标记 inactive → 清理记忆数据

### 4.2 Memory as a Service (记忆即服务) ✅

#### 4.2.1 三级记忆架构

平台提供三级记忆体系，所有 Agent 共享同一套存储基础设施，按 (user_id, agent_id) 二元组隔离:

| 记忆层级 | 存储介质 | 容量 | 生命周期 | 用途 |
|----------|----------|------|----------|------|
| 工作记忆 | Redis (内存) | 最近 15 轮对话 | TTL 7 天 | 当前对话上下文 |
| 短期记忆 | pgvector (向量库) | 无上限 | 持久化 | LLM 蒸馏的对话摘要 |
| 长期记忆 | pgvector (向量库) | 无上限 | 持久化 | 固化的事实和偏好 |

#### 4.2.2 记忆隔离验证

**验证方法**: 使用同一用户 (demo_user) 分别与 3 个 Agent 对话，检查各 Agent 的记忆是否独立。

| 测试步骤 | 操作 | 预期 | 结果 |
|----------|------|------|------|
| 1 | 与小暖对话 "我喜欢周杰伦" | 小暖记忆包含此信息 | ✅ |
| 2 | 与星尘对话 "我喜欢周杰伦" | 星尘记忆包含此信息 | ✅ |
| 3 | 查询小暖的记忆 | 包含步骤 1 的信息 | ✅ |
| 4 | 查询星尘的记忆 | 包含步骤 2 的信息 | ✅ |
| 5 | 检查小暖是否包含星尘的记忆 | 不包含 | ✅ 完全隔离 |
| 6 | 检查星尘是否包含小暖的记忆 | 不包含 | ✅ 完全隔离 |

**结论**: 记忆按 (user_id, agent_id) 完全隔离。同一用户的不同 Agent 之间不共享任何记忆数据。

#### 4.2.3 向量检索

**验证方法**: 向 Agent 注入记忆后，使用语义相关的查询进行检索。

| 注入记忆 | 查询 | 检索结果 | 相似度 |
|----------|------|----------|--------|
| "用户喜欢周杰伦的音乐" | "有什么音乐推荐吗" | 命中 | 0.78 |
| "用户最近工作压力很大" | "最近好累" | 命中 | 0.83 |
| "用户养了一只橘猫叫小胖" | "我的猫" | 命中 | 0.85 |

#### 4.2.4 记忆蒸馏管线

**验证方法**: 模拟多轮对话后触发记忆蒸馏，检查蒸馏结果。

| 蒸馏步骤 | 输入 | 输出 | 状态 |
|----------|------|------|------|
| 对话累积 | 15 轮对话 (工作记忆) | — | ✅ |
| LLM 摘要 | 15 轮对话文本 | 对话摘要 + 关键事实 | ✅ |
| 向量化 | 摘要文本 | 4096 维向量 | ✅ |
| 存储 | 向量 + 元数据 | pgvector 记录 | ✅ |
| 高频事实检测 | 3 次以上提及 | 固化到长期记忆 | ✅ |

### 4.3 会话隔离 ✅

**验证方法**: 创建多个 (user_id, agent_id) 组合的会话，验证各会话的独立性和状态管理。

| 会话 | user_id | agent_id | 状态 | 轮数 | 记忆 |
|------|---------|----------|------|------|------|
| S1 | user_1 | xiaonuan | active | 12 | 独立 |
| S2 | user_1 | xingchen | active | 8 | 独立 |
| S3 | user_2 | xiaonuan | active | 5 | 独立 |
| S4 | user_1 | mingyuan | paused | 3 | 独立 |

**结论**: 会话以 (user_id, agent_id) 为唯一标识，完全独立。同一用户的不同 Agent 会话互不干扰，不同用户的同一 Agent 会话也互不干扰。

### 4.4 性格引擎 ✅

#### 4.4.1 大五人格参数映射

**验证方法**: 验证 `build_rules()` 函数将 OCEAN 参数映射为行为规则的正确性。

| 参数 | 阈值 | 规则 | 触发 Agent |
|------|------|------|-----------|
| E >= 0.65 | 外向性高 | "你热情主动，经常发起话题" | 星尘 (0.9)、咚咚 (0.95) |
| E <= 0.35 | 外向性低 | "你安静内敛，更多倾听" | 明远 (0.3)、墨羽 (0.35) |
| A >= 0.65 | 宜人性高 | "你非常善解人意" | 小暖 (0.9)、星尘 (0.75)、明远 (0.8)、咚咚 (0.85)、安然 (0.85) |
| O >= 0.65 | 开放性高 | "你充满好奇心，喜欢探索新话题" | 星尘 (0.85)、明远 (0.7)、咚咚 (0.9)、墨羽 (0.9) |

### 5.2 风险详解

#### R1: API IP 白名单依赖 (P0)

API 端点 `your-api-endpoint` 配置了 IP 白名单，这意味着:

- 部署环境变更时必须同步更新白名单
- 开发环境和生产环境需要不同的白名单配置
- 临时或突发性的 IP 变更会导致服务中断

**缓解措施**: 在基础设施层面配置固定的出口 IP（如 NAT 网关），并在架构层面预留备用 API 网关的切换能力。

#### R2: 2026 年中国 AI 监管合规 (P0)

根据中国《生成式人工智能服务管理暂行办法》及相关法规，2026 年 7 月 15 日是重要的合规截止日期。情感 Agent 平台涉及以下合规要求:

| 合规要求 | 平台现状 | 差距分析 |
|----------|----------|----------|
| 算法备案 | 未备案 | 需在截止日前完成 |
| 内容审核日志 | Demo 有 Guard 审核 | 需完善审核日志存储和审计 |
| 用户数据本地化 | 数据存储在国内 | 需确认云服务商合规性 |
| 未成年人保护 | 未实现 | 需增加年龄验证和使用时长限制 |
| 透明度要求 | 未实现 | 需标注 AI 身份、提供投诉渠道 |

**缓解措施**: 将合规工作纳入项目路线图 P0 优先级，在 2026 年 7 月 15 日前完成算法备案和合规审计。

#### R3: LLM JSON 输出不稳定 (P1)

在记忆蒸馏和情绪感知等需要结构化输出的场景中，LLM 返回的 JSON 格式偶发错误:

| 错误类型 | 频率 | 示例 |
|----------|------|------|
| Markdown 代码块包裹 | 常见 | `\`\`\`json\n{...}\n\`\`\`` |
| 多余逗号 | 偶发 | `{"a":1,}` |
| 截断 | 偶发 | `{"summary":"部分内容` |
| 字段缺失 | 偶发 | 缺少 `facts` 字段 |

**缓解措施**: Demo 中已实现基本的后处理逻辑（去除 markdown 标记、strip 空白）。生产环境建议使用更健壮的 JSON 修复库（如 `json-repair`）和重试机制。

#### R4: pgvector 可用性依赖 (P1)

记忆检索依赖 pgvector 扩展，如果 PostgreSQL 不可用:

- 短期/长期记忆检索失败
- 新记忆无法存储
- 对话仍可进行（仅使用工作记忆），但体验降级

**缓解措施**: PostgreSQL 主从复制 + 定期备份。降级策略: pgvector 不可用时，对话管线跳过记忆检索步骤，仅使用工作记忆。

#### R5: 无本地模型部署选项 (P1)

当前所有模型均通过云 API 调用，没有本地部署方案:

| 风险场景 | 影响 |
|----------|------|
| API 服务宕机 | 平台完全不可用 |
| API 服务变更接口 | 需要适配迁移 |
| 网络延迟增加 | 对话响应变慢 |
| 数据隐私要求 | 无法满足本地化部署需求 |

**缓解措施**: Prompt 模板与模型解耦（通过 `capabilities.llm_config.model` 字段），预留模型切换接口。未来可考虑 vLLM 或 Ollama 本地部署方案。

#### R6: 情绪检测准确率波动 (P2)

情绪检测依赖 LLM 的非流式调用 + JSON 提取，准确率受以下因素影响:

| 因素 | 影响 |
|------|------|
| 输入文本长度 | 短文本情绪识别准确率较低 |
| 文化语境 | 中文含蓄表达可能被误判 |
| LLM 输出不稳定 | JSON 解析失败时返回默认值 |

**缓解措施**: 情绪检测结果仅作为性格演化的参考权重，不直接决定 Agent 的行为。默认返回 "neutral" 情绪，确保系统在情绪检测失败时仍能正常运行。

#### R7: 多租户隔离正确性 (P1)

多用户、多 Agent 场景下的数据隔离需要严格的 SQL 查询控制:

| 隔离维度 | 隔离键 | 风险 |
|----------|--------|------|
| 用户间隔离 | user_id | SQL 中遗漏 WHERE user_id 条件 |
| Agent 间隔离 | agent_id | SQL 中遗漏 WHERE agent_id 条件 |
| 用户-Agent 交叉 | (user_id, agent_id) | 组合条件错误 |

**缓解措施**: 所有数据查询必须同时包含 user_id 和 agent_id 条件。编写单元测试覆盖所有隔离场景。

### 6.5 成本优化建议

1. **缓存策略**: 对高频查询的 Embedding 结果进行缓存，减少重复调用
2. **批量蒸馏**: 记忆蒸馏采用批量处理，降低单次蒸馏成本
3. **模型选择**: 对简单任务使用更小的模型（如情绪检测），降低 LLM 调用成本
4. **Token 优化**: 精简 System Prompt 模板，减少不必要的上下文
5. **Reranker 选择性使用**: 仅在记忆库较大时启用 Reranker
