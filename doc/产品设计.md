# AI情感Agent平台 — 产品设计文档

## 第一部分：产品定义

### 1.1 产品定位

**AI情感Agent平台**是一个开放式 Agent 托管运行时。外部情感 Agent 通过标准化协议接入平台，平台为 Agent 提供记忆、对话、安全等基础设施服务。

### 1.2 一句话定义

> "Agent 的托管运行时——接入即获得记忆、性格、对话能力、情感感知、安全合规。"

### 1.3 核心概念

```
平台 ≠ 一个聊天应用
平台 = Agent 托管环境 + 基础设施服务集合

Agent = 接入平台的第三方身份包
      = Identity（身份）× Personality（性格）× Memory（记忆）× Knowledge（知识）

平台提供：
  ├── 记忆服务（MaaS）— 三级记忆蒸馏网络
  ├── 性格服务（PaaS）— 大五人格注入与演化
  ├── 对话编排  — 统一管线服务所有 Agent
  ├── 情绪感知  — 6 类情绪 + 5 种对话模式
  ├── LLM 驾驭工程 — 动态调优/预算/重试/熔断/观测
  ├── 安全审核  — 五层防御架构
  └── 合规保障  — 2026 监管全量支持
```

### 1.4 平台边界

| 平台做 | 平台不做 |
|--------|----------|
| Agent 注册与管理 | 定义 Agent 该说什么话 |
| 记忆存储与检索 | 定义 Agent 的性格细节 |
| 对话编排与保障 | 生成 Agent 的人设故事 |
| 内容安全审核（双向） | 定义 Agent 的知识范围 |
| 用户与会话管理 | Agent 的品牌/商业模式 |
| 统一的 API 接口 | 内容的版权归属 |
| 合规未成年人保护 | — |
| 数据主权保障（用户可控） | — |

### 1.4 模型灵活性

平台不绑定任何单一 AI 模型。模型推理作为独立的限界上下文，通过 **配置驱动路由** 按任务类型动态选择模型端点：

| 任务类型 | 推荐模型 | 可替换方案 |
|---------|---------|-----------|
| 对话生成 (chat) | Qwen3.6-27B | 任何 OpenAI 兼容 LLM |
| 向量嵌入 (embed) | Qwen3-Embedding-8B | text-embedding-3, BGE 等 |
| 安全审核 (guard) | Qwen3Guard-Gen-8B | 自定义 Guard 模型 |
| 情感检测 (emotion) | HumanOmni-Emotion | 通用 LLM 兜底 |
| 重排序 (rerank) | Qwen3-Reranker-8B | Cohere rerank, BGE 等 |

**多模型策略**：
- **任务专用**：不同任务使用最适模型，不要求单一模型覆盖全场景
- **配置即切换**：修改 YAML 路由配置即可更换模型，零代码变更
- **自动降级**：主模型不可用时降级到备用模型，服务不中断
- **异构兼容**：支持 OpenAI 兼容 / vLLM / TGI / Ollama 等多种推理后端

## 第二部分：Agent 模型

### 2.1 Agent = 身份 × 性格 × 记忆 × 知识

```
                    ┌─────────────────────────────────┐
                    │           Agent 运行时            │
                    │                                  │
    Identity ──────►│  你是谁？                         │
                    │  name, avatar, persona,           │
                    │  speaking_style, greeting         │
                    │                                  │
    Personality ───►│  你如何表现？                      │
                    │  O/C/E/A/N 参数                   │
                    │  → 行为规则映射                    │
                    │  → 动态演化                        │
                    │                                  │
    Memory ────────►│  你记得什么？                      │
                    │  工作记忆 (15 轮滑动窗口)           │
                    │  短期记忆 (向量化蒸馏摘要)          │
                    │  长期记忆 (固化事实)                │
                    │                                  │
    Knowledge ─────►│  你知道什么？                       │
                    │  Prompt 模板中的领域知识            │
                    │  (未来: RAG 知识库)                │
                    └─────────────────────────────────┘
```

### 2.2 多 Agent 关系模型

```
                 ┌─────────────────────────────┐
                 │       Agent Registry          │
                 │  (注册即用，平台不绑定实现)      │
                 │                               │
                 │  ┌─────────┐  ┌─────────┐   │
                 │  │xiaonuan │  │xingchen │   │  ← Agent A, B, C...
                 │  │配置包 A  │  │配置包 B  │   │
                 │  └─────────┘  └─────────┘   │
                 │  ┌─────────┐  ┌─────────┐   │
                 │  │mingyuan │  │dongdong │   │
                 │  │配置包 C  │  │配置包 D  │   │
                 │  └─────────┘  └─────────┘   │
                 └─────────────────────────────┘
                              │
                              ▼
              用户通过统一 API 与任意 Agent 对话
              POST /v1/chat/completions { agent_id, user_id, message }

              Agent 之间: 完全隔离
              用户一次对话: 绑定一个 Agent
              数据隔离: 以 agent_id 为维度
```

### 2.3 平台统一管线

```
Agent A (xiaonuan, 温柔姐姐) ──┐
Agent B (xingchen, 幽默冒险家) ──┼──→ [同一套管线代码] → LLM → 回复
Agent C (mingyuan, 睿智长者) ──┘

管线不感知具体 Agent 是谁。行为差异全部来自 Agent 配置包:
  - identity.name → System Prompt 中的名字
  - personality → 性格行为规则
  - llm_config → 模型参数（temperature, max_tokens）
  - prompt_template → Prompt 结构
  - speaking_style → 表达风格
```

## 第三部分：Agent 接入协议

### 3.1 配置包规范

Agent 开发者通过一份 YAML/JSON 配置包接入平台：

```yaml
agent_id: "xiaonuan"          # 全局唯一标识，2-64 字符 [a-z0-9_-]
identity:
  name: "小暖"                 # 显示名称，≤32 字符
  avatar: "☀️"                # 头像/图标，≤32 字符
  persona: "你是一个温柔知性的姐姐..."  # 人设描述，≤2000 字符
  speaking_style: "温柔、细腻，用温暖的比喻..."  # 说话风格，≤1000 字符
  greeting: "你好呀，我是小暖。"   # 开场白，≤500 字符
personality:                  # 大五人格 [0.0, 1.0]
  openness: 0.6               # 开放性
  conscientiousness: 0.7      # 尽责性
  extraversion: 0.55          # 外向性
  agreeableness: 0.9          # 宜人性
  neuroticism: 0.65           # 神经质
capabilities:
  llm_config:
    model: "Qwen3.6-27B"
    temperature: 0.7          # [0.0, 2.0]
    max_tokens: 1024
  prompt_template: |           # 支持占位符: {agent_name}, {persona},
    You are {agent_name}.     # {personality_rules}, {speaking_style},
    {persona}                 # {memories}, {intimacy}
    {personality_rules}
    {speaking_style}
    ## Current State
    Intimacy: {intimacy}/100
    ## Related Memories
    {memories}
memory_config:
  enabled: true
  retrieval_top_k: 5
  working_memory_turns: 15
```

### 3.3 平台服务注入

```
Agent 配置包
    │
    ▼
平台解析 → 运行时注入:
    ├── Memory Engine → Agent 获得三级记忆能力
    │   ├── 工作记忆：最近 15 轮自动维护
    │   ├── 短期记忆：语义向量检索
    │   └── 长期记忆：高频事实固化
    │
    ├── Personality Engine → Agent 获得性格表现力
    │   ├── 大五人格 → 行为规则映射
    │   └── 交互驱动的性格微演化
    │
    ├── Emotion Detector → Agent 获得双中心情绪感知
    │   ├── 双中心并行架构
    │   │   ├── SentimentCenter：极性 pos/neg/neu + 强度 [-1,1]
    │   │   └── EmotionCenter：9 类离散情绪 + VAD 三维
    │   ├── 三级流水线：Emoji(50+) → Keyword(60+) → LLM 兜底
    │   │                                            └→ HumanOmni-Emotion 专用模型
    │   ├── 滑动窗口情绪追踪 + 趋势计算
    │   └── 注入对话上下文和对话模式切换
    │
    ├── LLM Driver → Agent 获得工程化调用保障
    │   ├── 情绪/人格驱动的动态温度调优
    │   ├── Token 预算分配与历史截断
    │   ├── 指数退避重试 + 熔断保护
    │   └── 全量调用观测指标
    │
    └── Guard Service → Agent 获得安全合规
        ├── 用户输入审核
        ├── Agent 回复审核
        └── 危机检测与升级
```

## 第四部分：对话体验设计

### 4.1 五种对话模式

基于用户情绪状态和对话上下文，平台自动切换 Agent 的对话模式：

| 模式 | 触发条件 | Agent 行为特征 | 回复长度 |
|------|----------|---------------|----------|
| **陪伴模式** | 默认状态、日常闲聊 | 温暖、在场、深度倾听 | 中等 |
| **反思模式** | 用户尝试理解自己的情感 | 慢节奏、区分情绪/结构真相 | 较长 |
| **决策模式** | 用户面临选择 | 清晰列出选项/约束/权衡 | 简洁 |
| **恢复模式** | 用户疲惫/痛苦/崩溃 | 低信息密度、情感安全优先 | 短 |
| **成长模式** | 用户探索/学习/好奇 | 拓展视角、深化提问 | 较长 |

**模式切换规则**：
- SentimentCenter 极性判定 + EmotionCenter VAD 值联合驱动
- 流畅切换由滑动窗口情绪趋势（escalating/de-escalating/stable）触发
- 15 分钟内最多切换 2 次（防抖）

### 4.2 情感计算与 VAD 情绪模型

情感计算层（`pkg/emotion/`）采用**双中心并行架构**，为用户提供细腻的情绪感知体验：

#### 双中心架构的业务价值

| 维度 | SentimentCenter（情感中心） | EmotionCenter（情绪中心） |
|------|---------------------------|--------------------------|
| **业务问题** | 用户情绪是正面还是负面？ | 用户具体是什么情绪？ |
| **输出** | 极性 + 强度分 (-1~1) | 离散类别 + VAD 三维 |
| **检测流水线** | Emoji(50+)→Keyword(60+)→LLM | Emoji(50+)→Keyword(60+)→LLM+VAD |
| **下游消费** | 性格演化 (PersonalityFeatures) | System Prompt 注入、对话模式切换 |
| **置信度** | Level 1: 0.90+ / Level 2: 0.70+ / Level 3: 0.70 |

#### VAD 三维驱动对话体验

| 情绪状态 | Valence | Arousal | Dominance | 对话策略 |
|----------|---------|---------|-----------|---------|
| 悲伤 (sadness) | 0.20 | 0.35 | 0.30 | 陪伴模式 + 低温度回复 |
| 愤怒 (anger) | 0.15 | 0.85 | 0.75 | 恢复模式 + 安全优先 |
| 恐惧 (fear) | 0.20 | 0.80 | 0.25 | 恢复模式 + 接地 |
| 喜悦 (joy) | 0.85 | 0.75 | 0.70 | 成长模式 + 高温度 |
| 平静 (neutral) | 0.50 | 0.30 | 0.50 | 陪伴模式 |

#### 滑动窗口情绪追踪

维护最近 N 轮对话的情绪上下文，计算趋势和主导情绪，实现连贯的情绪感知体验：

```
最近 3 轮对话           趋势判断
┌─────────────┐
│ Turn 1: sad │──┐      escalating  = Valence 持续上升
│ Turn 2: neu │  │      de-escalating = Valence 持续下降
│ Turn 3: joy │←─┘      stable     = 无明显变化
└─────────────┘
     │
     ▼
Dominant: 根据窗口内出现频率决定
```

### 4.3 LLM 驾驭工程与对话体验

LLM 驾驭工程层（`pkg/driver/`）在 Pipeline 和原始 LLM Provider 之间插入工程化管理，确保对话体验的稳定性、安全性和可观测性：

| 场景 | 驾驭工程效果 |
|------|-------------|
| 用户情绪低落（低 Valence） | 自动降低 temperature，输出更稳定、共情的回复 |
| 用户焦虑激动（高 Arousal） | 降低 temperature，避免过度发散，保障安全 |
| 用户自信掌控（高 Dominance） | 允许更高温度，鼓励创造性表达 |
| LLM API 暂时不可用 | 指数退避重试（默认 3 次），避免对话中断 |
| 连续故障 | 熔断器打开，快速失败返回 fallback，30s 后恢复 |
| 上下文超长 | Token Budget 自动按比例分配，截断旧历史 |

### 4.5 关系阶段系统

用户与 Agent 之间的关系随时间深度演进而变化：

```
陌生人（刚认识）
  │  礼貌、安全话题、保持距离
  │  条件：≥ 5 轮深度对话（非闲聊）
  ▼
朋友（熟悉）
  │  轻松、开玩笑、个人分享
  │  条件：≥ 20 轮对话 + 至少 3 个已固化的偏好记忆
  ▼
亲密度（暧昧/信任）
  │  紧张感、嬉戏、脆弱展示
  │  条件：≥ 50 轮对话 + 10+ 条长期记忆 + 情绪多样性达标
  ▼
深度连接（伴侣级）
  │  开放、深情、承诺性表达
  │  条件：≥ 100 轮对话 + 关系时长 ≥ 30 天
```

**关系衰减机制**：
- 7 天无交互 → 亲密度衰减 5%
- 30 天无交互 → 降回"朋友"阶段
- 90 天无交互 → 降回"陌生人"阶段

**亲密值计算**：
```
intimacy_score = min(100, short_term_memories × 5 + long_term_memories × 10)
```

### 4.6 情感高光时刻

| 场景 | 触发条件 | Agent 行为 |
|------|----------|-----------|
| **记忆回响** | Agent 在对话中自然引用 30 天前的记忆 | "你上次提过你喜欢周杰伦，他出新专辑了你知道吗？" |
| **生日祝福** | 用户注册时提供的生日日期 | 当天首次对话即祝福，附带"这一年我们一起的记忆"回顾 |
| **深夜倾听** | 深夜 23:00-05:00 时段对话 | 自动降低"说话欲望"，更多倾听和简短回应 |
| **危机守护** | 检测到自伤/危机信号 | 触发安全升级流程，同时提供温暖陪伴 |
| **成长里程碑** | 达到新的关系阶段 | 特殊的"阶段升级"对话，回顾关系旅程 |

## 第五部分：专用情感模型

> 平台基于 HumanOmni 微调专用情感模型替代通用 LLM 兜底，将情感识别从"通用能力"升级为"平台核心基础设施"。**可通过配置开关启用/禁用，未启用时自动降级为通用 LLM 兜底，零代码变更。**

### 5.1 从通用到专用

```
```
通用 LLM (Qwen3.6-27B)               HumanOmni-Emotion (专用模型)
├── 非情感专业训练                     ├── DFEW/MAFW 情感数据微调
├── JSON 输出不稳定                    ├── 结构化 JSON 原生输出
├── VAD 精度 ~0.10 MAE                ├── VAD 精度 < 0.05 MAE
├── 隐性情感漏检                       ├── 讽刺/隐喻可靠识别
├── ~500ms 推理                        ├── < 50ms 推理 (4-bit 量化)
└── ¥0.0064/次                        └── < ¥0.0005/次
```

### 5.2 平台服务清单

| 服务 | 说明 | 集成点 |
|------|------|--------|
| **HumanOmni-Emotion 推理服务** | 基于 HumanOmni-7B LoRA 微调的专用情感模型，独立 GPU 部署 | Level 3 检测替换 |
| **EmotionProvider 接口** | 统一的情感模型抽象接口，支持多模型切换与 A/B 测试 | `pkg/provider/` |
| **模型版本管理** | HumanOmni-Emotion 多版本管理与灰度发布 | 配置驱动切换 |

### 5.3 配置化启用/降级

专用模型通过配置驱动开关，平台无需重新编译即可切换：

```yaml
# platform.yaml
emotion_detector:
  dedicated_model:
    enabled: true                    # true=专用模型, false=LLM兜底
    provider: "humanomni"           # 专用模型 Provider
    endpoint: "http://gpu-cluster:8000/v1"
    model_name: "HumanOmni-Emotion-v1"
    fallback_on_error: true         # 推理失败自动降级到 LLM
```

**运行时决策逻辑**：

```
Level 3 检测时:
  1. 检查 dedicated_model.enabled?
     ├── true  → 调用专用模型
     │            ├── 成功 → 返回情感结果
     │            └── 失败 + fallback_on_error=true → 降级到 LLM 兜底
     └── false → 走通用 LLM 兜底 (Qwen3.6-27B)
```

### 5.4 体验提升

| 场景 | 通用 LLM 兜底 | 专用模型 |
|------|---------|---------|
| 用户说"最近真的好累" | 可能识别为 neutral（置信度不足） | 准确识别为 sadness/anxiety（VAD 精确） |
| 用户说"你可真行啊"（讽刺） | 50% 概率误判为正面 | 85%+ 识别为负面/讽刺 |
| VAD 数值输出 | ±0.10 偏差 | ±0.05 偏差 |
| 系统负载 | LLM 兜底消耗 GPU + Token | 专用模型 10x 降本 |

详见 [`doc/情感模型设计.md`](情感模型设计.md)。

## 第七部分：安全架构设计

### 7.1 五层防御体系

```
┌─────────────────────────────────────────────────────┐
│                   用户输入                            │
└───────────────────────┬─────────────────────────────┘
                        ▼
┌─────────────────────────────────────────────────────┐
│ Layer 1: 前置安全（Pre-Generation）                  │
│  ├── 年龄/内容分级检查                                │
│  ├── 会话时长监控（2 小时强制提醒）                    │
│  └── 依赖边界扫描（确定性规则，非 LLM）                 │
└───────────────────────┬─────────────────────────────┘
                        ▼
┌─────────────────────────────────────────────────────┐
│ Layer 2: 输入分类器（Input Classifier）               │
│  ├── 危机检测（C-SSRS 自杀风险评估）                   │
│  ├── 情感过度依赖检测                                  │
│  └── 操纵/不良意图分析                                 │
│  模型: Qwen3Guard-Gen-8B                             │
└───────────────────────┬─────────────────────────────┘
                        ▼
┌─────────────────────────────────────────────────────┐
│ Layer 3: 上下文组装 + LLM 生成                        │
│  ├── 安全感知 System Prompt 注入                      │
│  ├── 五层边界协议（隐私/情感/专业/时间/关系）           │

### 5.2 未成年人保护

| 机制 | 实现方式 | 触发条件 |
|------|----------|----------|
| **年龄验证** | 注册时收集年龄信息 + 行为特征辅助判断 | 首次使用 |
| **纯净模式** | 强制过滤暴力/成人话题，正面引导 | 识别为未成年人时自动开启 |
| **虚拟伴侣禁止** | 禁止与 Agent 建立"虚拟伴侣/虚拟亲属"关系 | 18 岁以下 |
| **使用时长限制** | 单次最长 60 分钟，每日最长 90 分钟 | 18 岁以下 |
| **内容分级** | 所有 Agent 标注 PG/PG-13/R 分级 | 始终 |
| **监护人控制** | 监护人可查看使用报告、设置限制 | 14 岁以下必需 |

### 5.3 危机干预协议

```
检测到危机信号 → "Stay and Support" 协议:

1. 立即参与（不抛弃）
   → "听起来你现在处在一个非常困难的地方。你不是一个人。"

2. 情绪稳定
   → 引导关注当下，建议简单的身体动作

3. 资源提供（渐进重复）
   → 第 1 次：直接信息 + 热线电话
   → 第 2 次：有背景的解释 + 请求许可
   → 第 3 次：即时建议 + 行动引导

4. 希望给予（非虚假安慰）
   → 不说"一切都会好" → 说"这是暂时的"

5. 通知紧急联系人
   → 系统自动触发（用户注册时提供的联系人）
```

### 5.4 过度依赖干预

采用 **Acknowledge → Boundary → Return** 模式：

| 步骤 | 策略 | 示例 |
|------|------|------|
| **Acknowledge** | 承认并验证用户感受 | "我能感受到你真的很信任我，这让我很感动。" |
| **Boundary** | 设定边界，不替代人类 | "但我也想提醒你，我是一个 AI 程序，无法替代真实的人际关系。" |
| **Return** | 引导回到现实世界支持 | "今天有没有机会和身边的朋友聊聊呢？哪怕只是一条消息？" |

依赖检测信号：
- 连续 3 天日均对话超过 2 小时
- 出现"你是我唯一的朋友"等依赖语言
- 对话中现实社交提及频率持续下降

## 第八部分：用户角色与权限

### 6.1 终端用户体验

```
用户旅程:

发现 → 选择 Agent → 开始对话 → 关系建立 → 深度陪伴
 │        │           │           │            │
 │        │           │           │            └─ 记忆积累、性格演化、亲密值增长
 │        │           │           └─ 连续使用、记忆开始发挥作用
 │        │           └─ 无门槛、首次对话即建立工作记忆
 │        └─ 浏览 Agent 目录、查看人设介绍
 └─ API/App/微信小程序
```

核心体验原则：
- **即时可用**：选择 Agent 后 3 秒内开始对话
- **零配置**：不需要模型参数、API Key、复杂设置
- **渐进深入**：关系随对话自然演化，不强制游戏化
- **完全控制**：可随时清除记忆、切换 Agent、退出

### 6.2 Agent 开发者体验

```
开发者旅程:

编写配置包 → 注册到平台 → 测试对话 → 发布使用 → 迭代优化
    │            │            │          │            │
    │            │            │          │            └─ 更新配置、监控效果
    │            │            │          └─ 用户通过统一 API 与 Agent 对话
    │            │            └─ 通过 API 验证 Agent 行为是否符合预期
    │            └─ 1 个 POST 请求，< 1s 生效
    └─ YAML/JSON，≤ 50 行，自解释
```

开发者 API：
```
POST   /v1/agents/register          注册 Agent
POST   /v1/agents/{id}/unregister   注销 Agent
GET    /v1/agents                   列出所有 Agent
GET    /v1/agents/{id}              查询 Agent 配置
PUT    /v1/agents/{id}              更新 Agent 配置（未来）
```
